# backlog-infra — Backlog

A running list of work to do. Mark items `- [x]` when done and move
them to the Done section, `- [ ]` when open, `- [~]` when in progress,
`- [?]` when blocked on a design decision the user needs to weigh in
on, and `- [!]` once you've written answers inline and want the worker
to retry. The autonomous worker (`backlog run`) picks `- [ ]` and `- [!]` items and skips `- [?]`; surface those
to the user when waking them.

Items parked in `## Thinking` are never picked by the daemon — move
them to `## Open` when ready for work.

## Thinking

> **Rationalization (2026-05-28) — read before promoting items to `## Open`.**
> Each item is tagged **(priority · wave)**.
> **Priority:** P1 = prevents unattended damage / makes the fleet observable / foundational unblocker · P2 = output quality + correctness · P3 = larger architectural or meta-maintenance.
> **Wave (build order, derived from dependencies):** W0 foundations → W1 primitives → W2 consumers → W3 orchestration → W4 lifecycle.
>
> **Shared primitives — build once, then the tagged items are thin consumers (don't implement 5 bespoke versions):**
> 1. **Pause mechanism** (scope `item | project | fleet` → heartbeat-only, with reason + until-condition): account-cooldown, per-project budget, per-item ceiling, fleet-freeze, pause/resume.
> 2. **Tick guard + abort/reopen/annotate** (a check that can veto a tick): per-tick verification, circuit breaker, diff-size, secret-scan, protected-path, post-completion review.
> 3. **One notify/event channel** (extend `agent-notify.json`): dead-man's-switch, fleet digest, brittle-contract canary (+ existing cooldown alert).
> 4. **Structured event log** → feeds metrics, alerting, digest, monitor.
> 5. **D1 telemetry bus + liveness** (COMMITTED) → underpins the account-cooldown signal, fleet-freeze flag, version-skew, live dashboard. Sequence early as an enabler.
> 6. **Test/CI harness** → gates safely changing the driver for everything else. Do first.
> 7. **Backlog integrity**: schema lint + snapshot + the completion-flag recording item (in `## Open`).
>
> **Resolved decisions (2026-05-28):** D1 telemetry bus = **committed**. Compaction = **Archive (shared `backlog-agent-compact.mjs`) is canonical** — align per-project scripts. Branch-per-item isolation = **rejected**, in favor of commit-to-main + the damage guards (diff-size, verification, protected-path).

- [ ] **(P1 · W0) Test harness + CI for the driver itself.** `bin/backlog-agent` (~1300 lines bash) and `bin/backlog-agent-status.mjs` run the whole fleet with zero automated tests — every incident so far (unconditional `git add COOLDOWN_FILE`, BSD-awk `-i inplace` silent no-op, the `reclaim_stale_claims` runaway, the `pipefail` crash) is the class a test suite catches. Add: (a) a **bats** harness that spins up two throwaway clones of a scratch repo as fake "M1"/"M3" and asserts the §5 invariants — claim CAS picks one winner + loser `reset --hard`s, TTL reaper frees `[~]` after 90m, cooldown short-circuits to heartbeat, reclaim-before-claim keeps a non-zero diff; (b) **shellcheck** on `bin/*` in CI (would have flagged the pipefail/quoting bugs); (c) a `--inject-fault` mode (kill mid-claim, simulate rejected push) to actually exercise the §10 recovery paths instead of trusting them. → unblocks safe work on every other driver item.
- [ ] **(P1 · W0) Structured (JSON) event logs.** The monitor greps freeform log *prose* for failure strings, which is brittle. Emit structured lines for key events (`tick_start`, `claim`, `claim_lost`, `claude_exit`, `cooldown_armed`, `status_ok`/`status_fail`). Makes the monitor's matching robust and unlocks real metrics instead of regex-on-text. → enables: dead-man's-switch alerting, efficiency metrics, fleet digest.
- [ ] **(P1 · W0) Backlog schema validation / linter.** The queue is hand- and agent-edited markdown, and that causes breakage — e.g. "web dashboard item back to `[?]` (agent flipped it)" and the `auto_flip_blocked` `[user]`-at-line-start bug. Add a validator (git pre-commit hook or top-of-tick guard) asserting: sections in canonical order, every item a valid marker, no item in two sections, no duplicate titles, `[~]` items have an inflight owner. Makes the markdown queue robust to malformed edits instead of trusting every writer.
- [ ] **(P1 · W0) Backlog snapshot safety net.** The backlog is the single source of truth, mutated by autonomous agents — and sacred-geography's `docs/Backlog.md` dropped 2,033 lines in one pull. Git is the history, but an explicit periodic snapshot (daily tag, or off-repo copy) makes "an agent nuked a backlog" a one-command restore instead of git archaeology. Cheap insurance.
- [ ] **(P1 · W1) D1 telemetry bus + decouple liveness from commits (COMMITTED — enabler, sequence early).** Riding ephemeral status (heartbeat, tokens, liveness) on commits to shared `.claude/` files is the root cause of a whole incident class: phantom tokens, orphaned-stash churn, per-host file conflicts, the untrack sweep. Finish the **Path B (D1-backed realtime health ingest)** migration started in backlog-dashboard: git keeps only durable, mergeable state (`docs/Backlog.md`, work commits, the per-tick ledger); telemetry pushes to D1. **Critically:** `reclaim_stale_claims`'s liveness signal is "a status commit in the last 90m" — exactly what broke today, so a failed status hook makes the reaper unable to tell a live-but-not-committing daemon from a dead one (correctness hazard). Moving liveness onto the independent D1 heartbeat fixes the reaper too. **Free-tier constraint (must stay on CF free):** D1 free limits are 100k rows written/day, 5M rows *read*/day, 5 GB storage (verified 2026-05-28, [pricing](https://developers.cloudflare.com/d1/platform/pricing/)). At ~1,500 ticks/day fleet-wide, writes (~4.5k/day), requests, and storage are all <5% of the ceiling — the only blow-up risk is **read-row amplification**, since D1 bills rows *scanned*, not returned: an unindexed dashboard poll over a growing history table on a short refresh loop can hit 100s of millions of reads/day. Design to avoid it: (a) a tiny **current-state table** (~1 row per project×host) for the live dashboard view, with the append-only history reserved for analytics; (b) **index history** on `ts`/`project`/`host`; (c) **push (SSE/WebSocket) over polling** from the ingest Worker; (d) a **retention/rollup** job on history. → enables: account-cooldown signal, fleet-freeze flag, version-skew, live dashboard.
- [ ] **(P1 · W1) `backlog-agents doctor` — preflight health command.** Codify the environment checks we run by hand when the fleet misbehaves: double-daemons from old labels, plists pointing at canonical (de-symlinked) paths, gitignore patterns present, status-hook resolves, budgets file present, daemon count matches the manifest. One command that reports drift — turns "why is the fleet weird" into a 2-second check.
- [ ] **(P2 · W1) Driver version-skew visibility + on-demand fleet sync.** M1 ran the broken driver for hours purely because the fix was on origin but unpulled, and `backlog-agents daemon-sync` is on a 24h cycle — and because `bin/` is *shared*, a fix isn't live until every machine pulls. Make "is everyone running the fix?" a glance: (a) stamp each tick's status with the **git SHA of `backlog-infra/bin`** the daemon is running; the dashboard surfaces skew (`M1 abc123 · M3 def456 · origin def456 ⚠`); (b) add an on-demand fleet sync trigger so deploying a driver fix doesn't wait up to 24h. → feeds: fleet-freeze (skew-zero is its clear condition).
- [ ] **(P1 · W1) Prompt-injection / malicious-backlog threat model.** Daemons run `claude -p` with broad shell/commit permissions and the *backlog item text is the prompt* — a poisoned item (via a compromised repo, a dependency README, or a synced file) could instruct an agent to exfiltrate secrets or push malicious code, unattended. Everything else is output-side (secret-scanning, diff guards); nothing vets the *input*. Define a trust boundary on backlog additions (who/what may append `[ ]` items) and a constrained tool policy for autonomous mode. Highest-value security gap.
- [ ] **(P1 · W2) Per-tick outcome verification.** [guard-primitive consumer] `tick_once` treats `claude exit 0` + claude's self-report as success, but exit 0 doesn't mean the gate is green. After claude exits 0, the daemon independently confirms a `work_commit` actually landed AND the project's gate (`npm test`/typecheck) passes on it — if not, unclaim + reopen rather than recording silent "progress." Distinct from the completion-flag *recording* item in `## Open` — this *enforces*. → pairs with acceptance-criteria (gives it a concrete target).
- [ ] **(P2 · W2) Structured "done-when" acceptance criteria per item.** Items are freeform prose, which gives the verifier nothing concrete to assert and is why "agent thinks it's done but isn't" happens. An optional `done when:` field gives per-tick verification a real target and tightens the definition of done.
- [ ] **(P2 · W2) Post-completion review pass.** [guard-primitive consumer] After an item is marked done, a cheap second agent (haiku) diffs the `work_commit` against the item description and flags intent-vs-change mismatches *before* completion is accepted. Distinct from per-tick verification (which runs the build gate); this reviews semantic intent.
- [ ] **(P1 · W2) Item-level circuit breaker / retry budget.** [guard-primitive consumer] The "Daemon broken" item was claimed/unclaimed 6× in one window. Track a per-item failure counter; after K failed attempts (claude exit ≠0, or failed verification), stop re-claiming, auto-flip to `[?]` with the failure log attached, and surface it. Converts a silent token-burning retry storm into one actionable blocker.
- [ ] **(P1 · W2) Diff-size guardrail on work commits.** [guard-primitive consumer] Flag/pause any tick whose `work_commit` touches more than N files or M lines unless the item opts in — exactly the guard that would have caught sacred-geography's 2,033-line backlog deletion. Change-*size* guard; complements the failure-*count* circuit breaker.
- [ ] **(P1 · W2) Secret-scanning pre-push gate.** [guard-primitive consumer] A `gitleaks`/`trufflehog` hook on autonomous commits so a daemon can never push a key/token. Important when agents commit unattended to repos that may be public.
- [ ] **(P2 · W2) Protected-path list.** [guard-primitive consumer] Files like the driver `bin/backlog-agent`, `release.mjs`, `CLAUDE.md`, `.gitignore` should never be modified by an autonomous tick without flagging for review — stops an agent from silently editing its own controller. → also the path-based signal fleet-freeze uses to classify a change as fleet-affecting.
- [ ] **(P1 · W2) Per-item token/turn ceiling.** [pause-primitive consumer, scope=item] A single hard item can spiral (many turns, huge spend) within one tick, below the per-project budget's radar. Abort and reopen-with-a-note when one item exceeds X tokens/turns — catches the runaway *task*, not just the runaway *project*.
- [ ] **(P1 · W2) Per-project budget *enforcement* (not just warnings).** [pause-primitive consumer, scope=project] `backlog-budgets.json` today only colors the dashboard when a target is exceeded. Make it enforcing: if a project blows a per-project token cap in a rolling window, its daemon auto-pauses to heartbeat-only and alerts. Distinct from the account-level *plan-limit* cooldown (Anthropic's cap); this is self-imposed *spend* control at per-project granularity.
- [ ] **(P1 · W2) Account-level shared cooldown.** [pause-primitive consumer, scope=fleet; rides the D1 bus] The Anthropic plan limit is **per-account**, shared across every machine and project — but `start_cooldown` arms `.claude/agent-cooldown.json` *per-project, per-machine*. So hitting the limit makes all 7 projects on both machines each fail-loop independently. Introduce a single account-level cooldown signal (D1 key, per the telemetry-bus item) that `tick_once` consults first, so the **whole fleet stands down at once** the moment any daemon detects the plan limit.
- [ ] **(P1 · W2) Dead-man's-switch alerting — proactive, not passive.** [notify-channel consumer] Today the cooldown-add bug *knew* it was failing (`fatal: ... did not match any files` every tick) and the dashboard *showed* STALE, but nothing escalated to the user for ~8h. Fire the notify channel when a daemon's last *successful* status commit is older than ~2× `HEARTBEAT_SECONDS`, OR when the status hook's git ops fail N consecutive ticks. Promote "STALE-too-long / repeated-hook-failure" from a dashboard color into an actual push.
- [ ] **(P1 · W2) Automated fleet digest + remote unblock.** [notify-channel consumer] Auto-generate the "what each project accomplished, cost, completions" report on a daily/weekly cadence to Slack/email — and let the user answer `[?]` blocked items by replying to the notification, closing the human-in-loop gap remotely. → depends on: efficiency metrics, notify channel.
- [ ] **(P2 · W2) Efficiency metrics from the per-tick ledger.** The JSONL ledger is only surfaced as burn rate today. Roll it into decision data: cost per completed item, success rate per project, tokens-per-completion trend, time-to-completion, retry rate. Answers "which projects are expensive per item?" and "which items keep failing?". → depends on: the completion-flag recording item in `## Open`, structured logs.
- [ ] **(P2 · W2) Guard the brittle external contracts.** [notify-channel consumer] Cooldown detection is **regex-matching Anthropic's error text** — if they reword "usage limit reached," cooldown silently stops working and the fleet returns to fail-loops. Same fragility with `claude` CLI flags and `jq`/`gawk`/`fswatch`/`wrangler` versions. A scheduled canary test asserting "the plan-limit signature still matches a known sample" and "claude still accepts our flags" catches upstream changes before they break the fleet silently.
- [ ] **(P2 · W3) Fleet freeze on in-flight driver changes (infra = highest priority).** [pause-primitive consumer, scope=fleet; composite — depends on version-skew, canary, priority-scheduling, protected-path, D1 freeze-flag] Because `bin/` is *shared*, a change to the machinery affects every project — today's cooldown bug silently broke all 7 daemons via stale infra. When a *fleet-affecting* infra change is in flight, other projects pause rather than work on stale infrastructure. **Gate on deployment freshness, NOT on "infra backlog empty"** (the infra backlog never empties — that rule would starve the fleet forever): (a) distinguish *fleet-affecting* changes (touch `bin/`, the status hook, the tick contract — via the protected-path signal) from infra *feature/doc* work which must NOT pause anyone; (b) on a fleet-affecting change, set a fleet-wide **"infra freeze"** flag (D1 key) every daemon checks at top-of-tick → heartbeat-only while set; (c) auto-clear once version-skew is zero AND a canary tick passes; (d) backlog-infra ticks **highest priority**. **Critical gotcha:** backlog-infra's OWN daemon must be EXEMPT — else the daemon that has to land + verify the fix is itself frozen → permanent deadlock. Manual override always available.
- [ ] **(P2 · W3) Canary rollout for driver changes.** Because `bin/` is shared, one bad driver commit breaks all 7 projects at once — which is what the cooldown-add bug did. Before a driver change goes fleet-wide, run it against one **canary project** for a cycle and assert health, then let the rest pull. Blast-radius control. → the "canary passes" half of the fleet-freeze clear-condition.
- [ ] **(P2 · W3) Priority-weighted fleet scheduling.** All 7 daemons tick independently and compete first-come for the shared account budget. Add a project priority so high-value work gets more frequent ticks / wins contention; the elegant version is a single fleet scheduler allocating the shared token budget by priority rather than 7 independent loops. → underpins the infra=highest-priority half of fleet-freeze.
- [ ] **(P2 · W3) Fleet pause / stop / resume control (all machines).** [pause-primitive consumer, scope=fleet, manual] Pause = suspend tick loop without losing claim/state; stop = full halt; resume/restart = bring back up — uniformly over every machine, not just the local one. Sibling of fleet-freeze (shares the pause mechanism + a cross-machine flag).
- [ ] **(P2 · W4) New-machine bootstrap drill + incident runbook.** If a machine's disk dies, how fast can a replacement be stood up? The manifest + `kash_setup.sh` exist, but an actually-*tested* "bare machine → fully-running fleet" drill — plus a written incident runbook (symptom → check → fix, codifying what was diagnosed by hand during the cooldown incident) — turns recovery from archaeology into a checklist. `doctor` is the automated half; this is the human half.
- [ ] **(P3 · W4) Dead-project / archive-candidate detector.** Some projects' backlogs may be effectively dead — no open items for weeks, just burning heartbeat cycles and dashboard space. Auto-flag projects that should move to `archived/` (ties to `dev-projects archive`), keeping the active fleet lean.
- [ ] **(P3 · W4) Weekly backlog-hygiene self-audit tick.** A scheduled agent that sweeps each backlog for stale items, duplicates, things parked in Thinking forever, and mis-marked states, then proposes cleanups — meta-maintenance so the queue doesn't rot.
- [ ] **(P3 · W4) Align compaction on the Archive strategy (decided).** Canonical = the shared `bin/backlog-agent-compact.mjs` (Done → Archive); `architecture.md` §6/§10 already document this. Replace per-project *title-only* variants (e.g. sacred-geography's `scripts/compact-backlog.mjs`) with the shared behavior so every project compacts identically. → unblocks the trigger fix below.
- [ ] **(P3 · W4) Fix the auto-compact trigger.** `_post_tick_cleanup` gates the compact call on `done_count > 20`. Under the Archive strategy this clears correctly (compacted items leave `## Done`), so the fix is: remove the title-only special-casing and confirm the count actually drops post-compact once all projects use Archive. Depends on the alignment item above.
- [ ] **(P3 · parked — scale only) Per-item claims to kill the single-file CAS bottleneck.** Every claim/unclaim/reclaim is a full-file CAS edit on one `docs/Backlog.md`, and the awk-inplace footgun lives there. Fine at 2 machines / 7 projects; if the fleet grows, an append-only claim log or one-file-per-item would cut contention and eliminate the file-rewrite class. Not now.
